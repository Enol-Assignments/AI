# 需求分析

## 1. 项目概述

本项目为一款运行于微信小程序的双人对战游戏。玩家（或 AI 代理）在随机生成的迷宫场景中操控角色，通过走位、射击与技能使用击败对手。所有算法均在小程序前端本地运行，无独立后端服务。游戏每局随机生成地图，保证可重玩性。

---

## 2. 技术架构说明

本项目采用纯前端架构，所有模块均在微信小程序客户端执行：

- **迷宫生成、路径规划、行为树决策**：使用 JavaScript 实现，直接运行于小程序逻辑层。
- **游戏渲染**：使用微信小程序 Canvas API 进行 2D 绘制。
- **无后端**：不部署独立服务器，所有状态由本地维护。

---

## 3. 环境设计

**地图结构**：基于二维网格（Grid）构建，保证全连通性（任意两点间至少存在一条可行路径）。

**迷宫生成**：采用随机深度优先搜索（DFS）或 Prim 算法生成迷宫骨架，并引入"破墙"机制产生回路，提升地形复杂度与对战策略空间。每局开始时重新生成地图，确保路径规划具有随机性与挑战性。

**掩体系统**：地图包含固定墙壁与可破坏掩体。掩体同时作为射线检测（Raycasting）的遮挡物，影响视野判定与射击逻辑。

---

## 4. 角色系统

角色具有差异化属性，支持不同战术风格（肉盾型 / 爆发型等）。

| 属性 | 说明 |
|---|---|
| 生命值 HP | 归零判负；超时触发狂暴机制（持续扣血，强制收尾） |
| 攻击力 ATK | 实际伤害受对方防御力修正；弹夹有限，固定速率自动装填 |
| 防御力 DEF | 按比例抵消弹药伤害；特定状态可获额外护盾 |

**技能列表**

- **反弹磁场**：子弹碰墙后按法线向量物理反射，增加攻击不可预测性。
- **算法伤害**：从 0–9 随机抽取三个数相乘，结果作为下一发的攻击力。
- **时间穿越**：开局生成随机血量无敌分身，本体血量降至与分身相同时本体消失。
- **乒乓球手**：发射可持续反弹的弹球，发射间隔较长，发射期间可反弹对方来袭弹药。
- **布朗运动**：发射随机游走小球，直至命中对手，不可被反弹。

---

## 5. AI 系统（核心算法设计）

AI 行为分为三个模块，均在小程序本地运行，按需以低频调用外部 LLM API。

### 5.1 迷宫生成模块

| 项目 | 内容 |
|---|---|
| **解决的问题** | 自动生成结构合法、布局随机、具有一定复杂度的对战地图 |
| **方法选择理由** | 规则直接手写地图无法保证每局差异性；DFS/Prim 能在保证全连通的前提下高效随机生成，且破墙参数可控，便于调节地图复杂度 |
| **所在模块** | 游戏初始化模块（`maze/generator.js`），每局开始前执行一次 |
| **输入 / 输出** | 输入：网格尺寸、破墙概率；输出：带墙体与掩体标记的二维地图矩阵 |

### 5.2 路径规划模块（A\* 算法）

| 项目 | 内容 |
|---|---|
| **解决的问题** | AI 角色在迷宫中实时计算到目标点的最优路径（"去哪里"） |
| **方法选择理由** | 相比 BFS 纯广度搜索，A\* 引入曼哈顿距离启发函数，显著减少搜索节点数，满足实时游戏的性能要求；相比硬编码规则，能自动绕过动态障碍 |
| **所在模块** | AI 移动控制模块（`ai/pathfinding.js`），每次目标切换或地图变更时重新触发 |
| **输入 / 输出** | 输入：当前地图矩阵、AI 坐标、目标坐标；输出：从当前位置到目标的最短路径节点序列 |

**行为切换逻辑**：HP 高于阈值时以敌方坐标为目标（进攻）；HP 低于阈值时切换为撤退模式，寻找距敌最远且有掩体遮挡的安全区坐标。

### 5.3 战术决策模块（行为树 + 大模型 Agent）

#### 底层行为树（本地实时决策）

| 项目 | 内容 |
|---|---|
| **解决的问题** | 射击时机判定与子弹规避等高频实时行为（"做什么"） |
| **方法选择理由** | 行为树结构清晰、状态可控，适合实时高频决策；在有限状态空间下表现稳定，且无需训练数据 |
| **所在模块** | AI 实时控制模块（`ai/behaviorTree.js`），每帧或固定时间步执行 |
| **输入 / 输出** | 输入：视野内敌方位置、子弹轨迹向量、双方血量与弹药量；输出：当前帧行动指令（移动方向、是否开火、是否闪避） |

具体行为：
- **射击判定**：Line-of-Sight 射线检测，敌方在射程内且路径无不可破坏障碍时触发开火。
- **子弹规避**：实时监测敌方弹药轨迹，预测碰撞时间 $t$，触发侧向闪避。

---

## 6. 技术选型小结

| 模块 | 方法 | 类别 | 运行位置 |
|---|---|---|---|
| 地图生成 | 随机 DFS / Prim + 破墙 | 传统算法 | 本地 JS |
| 路径规划 | A\*（曼哈顿启发） | 启发式搜索 | 本地 JS |
| 实时决策 | 行为树（Line-of-Sight + 碰撞预测） | 规则 AI | 本地 JS |
| 游戏渲染 | 微信小程序 Canvas 2D | — | 本地渲染 |

# 开发文档

## 1. 系统目录结构

为了将“UI 渲染”与“底层算法”严格剥离，我们在微信默认目录的基础上新增 `game_core` 目录。包含游戏主要算法和逻辑。

```ASCII
.
|   app.js
|   app.json
|   ... (微信默认配置文件)
|
+---game_core                <-- 【新增】游戏核心引擎与算法库（纯 JS，无 UI）
|   |   constants.js         <-- 全局常量与接口定义（充当 C++ 的头文件）
|   |
|   +---maze                 <-- 迷宫生成子模块
|   |       generator.js     
|   |
|   +---ai                   <-- 战术与寻路子模块
|   |       pathfinding.js   
|   |       behaviorTree.js  
|   |
|   \---engine               <-- 游戏状态管理与物理结算
|           gameLoop.js      
|           physics.js       
|
+---components               <-- 微信自定义组件（如血条、摇杆等）
|
\---pages                    <-- 页面 UI 表现层
    \---index
            index.js         <-- 仅负责接管用户输入和控制 Canvas 绘图
            index.wxml
```

## 2. 接口定义规范（替代头文件方案）

在没有任何类型提示的 JS 中协作，如果不定义好输入输出，合并代码时绝对会灾难。我们统一在 `game_core/constants.js` 中使用 JSDoc 格式来模拟“结构体”和“头文件声明”。所有组员在编写函数前，必须严格对齐这里的数据结构。

**全局地图数据结构标准（MapGrid）：**
* 约定二维数组 `map[y][x]`，其中 `0` 表示可通过的空地，`1` 表示不可破坏的墙壁，`2` 表示可破坏掩体。

**角色状态数据结构标准（EntityState）：**
* 包含坐标 `{x, y}`、生命值 `hp`、当前动作状态 `action` 等。

## 3. 模块拆解与开发分工

### 3.1 迷宫生成引擎 (`game_core/maze/generator.js`)
* **功能定位**：游戏开局前的高负载计算模块，负责吐出一张合法的二维地图。
* **接口规范**：
    需要暴露一个核心函数 `generateMaze(width, height, breakRate)`。外部传入地图的宽高和破墙概率，该模块在内部跑完 DFS 算法后，返回一个符合 `MapGrid` 标准的二维数组。
* **开发注意**：内部逻辑随便折腾，只要保证吐出来的矩阵是全连通的就行。

### 3.2 AI 寻路算法 (`game_core/ai/pathfinding.js`)
* **功能定位**：底层驱动引擎，解决“如何避开障碍到达目标”的纯数学问题。
* **接口规范**：
    需要暴露函数 `findShortestPath(map, startPos, targetPos)`。接收当前的二维地图矩阵和起终点坐标，返回一个包含下一步移动坐标的数组 `[{x1, y1}, {x2, y2}...]`。
* **开发注意**：这是纯粹的 A* 算法实现。不用管角色是谁，只负责算最短的格子路径。

### 3.3 战术决策中枢 (`game_core/ai/behaviorTree.js`)
* **功能定位**：AI 玩家的大脑，负责在每一帧判断当前该干什么（开火、走位还是发呆）。
* **接口规范**：
    暴露函数 `tickBehaviorTree(aiState, playerState, bullets)`。输入当前战场上所有实体的状态快照，输出当前帧 AI 应该执行的指令，例如返回枚举值 `ACTION_SHOOT` 或 `ACTION_MOVE_LEFT`。
* **开发注意**：里面需要实现 Line-of-Sight（视线检测）来判断玩家是否在射击直线上，以及简单的向量运算来预测子弹碰撞。

### 3.4 游戏主循环与渲染 (`pages/index/index.js`)
* **功能定位**：整个游戏的心脏起搏器和显示器。
* **接口规范**：
    利用小程序的 `requestAnimationFrame` 或 `setInterval` 建立一个基础时间步（比如 16ms）。在这个循环里，依次调用战术中枢获取指令、更新物理坐标，最后调用 Canvas API 把 `MapGrid` 和所有角色画在屏幕上。
